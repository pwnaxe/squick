# Releasing Squick

This document describes the steps to ship a new version of Squick across
all supported distribution channels.

## One-time setup

### crates.io

1. Create an account at <https://crates.io>.
2. Generate an API token (Account Settings -> API Tokens).
3. Run `cargo login` locally and paste the token. CI uses
   the `CARGO_REGISTRY_TOKEN` GitHub Actions secret.

### npm

1. Create an account at <https://www.npmjs.com> and create the
   `hubhorizonllc` organization. The umbrella package publishes as
   `@hubhorizonllc/squick`; the unscoped `squick` name is already
   owned by an unrelated 2015 gulp plugin (`yourpalal`). All
   platform packages live under the same scope
   (`@hubhorizonllc/squick-<platform>`).
2. Generate an automation token (Access Tokens -> Generate New Token
   -> Automation).
3. Add it to the repository as the `NPM_TOKEN` secret.
4. The publish workflow uses npm provenance, which requires the
   workflow `id-token: write` permission (already set).

### PyPI

1. Create an account at <https://pypi.org> and reserve the `squick`
   project name.
2. Configure a trusted publisher under "Project -> Publishing":
   - Owner: `<org>`
   - Repository: `squick`
   - Workflow filename: `publish-pypi.yml`
   - Environment: `pypi`
3. In the repository, create a GitHub Environment named `pypi`. No
   secret is needed because trusted publishing uses OIDC.

### GitHub repository

1. Push the repository to `github.com/<org>/squick` (matching the
   `repository` field in every Cargo.toml).
2. Add `CARGO_REGISTRY_TOKEN` and `NPM_TOKEN` repository secrets.
3. Set a real legal contact email in `TRADEMARKS.md` (currently a
   `legal@horizon-llc.example` placeholder) before the first publish.

### Local tooling

- `cargo install cargo-dist --locked` (provides the `dist` binary).
  Currently pinned to 0.31.x in `dist-workspace.toml`.

## Release flow

### 1. Bump the version

Keep every version in lockstep:

- `Cargo.toml` (`[workspace.package].version`)
- `wrappers/npm/squick/package.json` and every
  `wrappers/npm/platforms/*/package.json` (the umbrella package's
  `optionalDependencies` is rewritten by CI from the tag, but local
  shells should still match)
- `wrappers/pypi/pyproject.toml`
- `extensions/vscode/package.json`

### 2. Verify locally

```bash
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
dist plan
```

`dist plan` must list five target archives, each containing the
`squick` binary, `LICENSE`, and `dictionaries/`.

### 3. Publish library crates to crates.io

Handled automatically by `publish-crates.yml` on tag push (parallel
to the binary release). The workflow publishes the five library
crates in dependency order with a 30 s pause between each step for
the crates.io sparse index to refresh.

To run it manually (e.g. for a dry-run preview) use:

```bash
gh workflow run publish-crates.yml -f dry_run=true
```

After the workflow completes, `cargo install squick-cli` works for
any Rust user. The published binary is also discoverable as
`squick`.

### 4. Cut a GitHub release

Push a SemVer tag matching the workspace version:

```bash
git tag v1.0.0
git push origin v1.0.0
```

This triggers three workflows in sequence:

1. `release.yml` (generated by `dist`) builds five binary archives,
   computes checksums, generates `squick-cli-installer.{sh,ps1}`, and
   creates a GitHub Release.
2. `publish-npm.yml` listens to `release: published`, downloads the
   archives, splices each binary into the matching
   `wrappers/npm/platforms/*` shell, and publishes every platform
   package plus the umbrella `squick` package to npm.
3. `publish-pypi.yml` similarly builds five platform wheels (one per
   target triple) and publishes them to PyPI via trusted publishing.

### 5. Verify a fresh install

On a clean machine (or in a container):

```bash
# crates.io
cargo install squick-cli && squick --version

# GitHub Releases (Unix)
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/<org>/squick/releases/download/v1.0.0/squick-cli-installer.sh | sh
squick --version

# GitHub Releases (Windows)
irm https://github.com/<org>/squick/releases/download/v1.0.0/squick-cli-installer.ps1 | iex

# npm (and MCP one-liner). The npm name "squick" was taken in 2015,
# so the umbrella package lives at @hubhorizonllc/squick. The binary
# on PATH is still plain `squick`.
npm i -g @hubhorizonllc/squick && squick --version
npx -y @hubhorizonllc/squick mcp   # runs the MCP server with zero install footprint

# PyPI
pip install squick && squick --version
python -c "import squick; print(squick.binary_path())"
```

The `dictionaries/` directory is shipped next to the binary in every
channel, so auto-discovery works without further configuration.

## Distribution channels

- **crates.io** — `cargo install squick-cli` (configured)
- **GitHub Releases** — `curl ... | sh` or `irm ... | iex` (configured)
- **npm** — `npm i -g @hubhorizonllc/squick` or `npx -y @hubhorizonllc/squick mcp` (configured; scoped because unscoped `squick` was taken in 2015)
- **PyPI** — `pip install squick` (configured)
- **VS Code Marketplace** — extension stub only, planned for 1.x

## Rolling back

`cargo yank` deprecates a published crate version without deleting
it. A yanked version is invisible to new resolves but still works
for existing lockfiles.

```bash
cargo yank --vers 1.0.0 -p squick-cli
```

For npm, use `npm deprecate` (preferred) or `npm unpublish` (only
allowed within 72 hours of publish).

```bash
npm deprecate squick@1.0.0 "Use 1.0.1 instead"
```

For PyPI, releases are immutable. To pull a broken release, yank it:

```bash
# Via web UI: Manage project -> Releases -> Yank
```

For GitHub releases, delete the release in the web UI and remove
the tag locally and remotely:

```bash
git tag -d v1.0.0
git push origin :refs/tags/v1.0.0
```
